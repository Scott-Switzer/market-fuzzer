# Research references and bounded use

This project is a fictional, deterministic education prototype. The references below
informed interfaces and safeguards; no substantial third-party code was copied and no
reference implementation is a runtime dependency.

| Source | What we learned and design used | Implemented in | Deliberately not integrated | Attribution / license |
| --- | --- | --- | --- | --- |
| [ABIDES](https://github.com/abides-sim/abides-jpmc), [paper](https://arxiv.org/abs/1904.12066) | Separate exchange actions, agent observations, decisions, and transport latency. Record each time boundary instead of treating latency as one decorative input. | `app/simulation.py`, `app/exchange/`, `app/execution_arena.py` | ABIDES kernel, protocols, agents, or datasets. | Architecture/paper citation only; no ABIDES code or assets copied, so its repository license adds no runtime obligation here. |
| [ABIDES-Gym](https://arxiv.org/abs/2110.14771) | A strategy-facing environment needs bounded observations and an action/result boundary. | `app/execution_arena.py`, `app/api/app.py` | Gym dependency and RL interface. | Paper attribution only. |
| [HftBacktest](https://github.com/nkaz001/hftbacktest), [documentation](https://hftbacktest.readthedocs.io/) | Feed, decision, entry, and cancel latency; partial fills; acknowledgments; and quantity ahead must be explicit rather than optimistic. | `app/simulation.py`, `app/exchange/`, replay evidence in `app/static/arena.js` | Market-data replay, Rust engine, venue models, and claims of a complete queue model. | Repository/documentation citation only; no code copied or dependency added. |
| [JAX-LOB](https://arxiv.org/abs/2308.13289) | Treat policy × worlds × seeds as a deterministic evaluation matrix. | `app/execution_arena.py` | JAX, GPU execution, or an order-book migration. | Paper attribution only. |
| [Stylized-fact realism metrics](https://arxiv.org/abs/1912.04941) | Mechanical checks do not establish real-market calibration. | `app/execution_arena.py`, `docs/LIMITATIONS.md` | Claims of calibration from a small diagnostic subset. | Paper attribution only. |
| [Implementation shortfall](https://arxiv.org/abs/1205.3482), [constant participation](https://arxiv.org/abs/1210.7608), [liquidity-aware tactics](https://arxiv.org/abs/1803.05690) | Separate completion, shortfall, participation, and liquidity-aware behavior. | `app/simulation.py`, `app/execution_arena.py` | Investment advice, real execution estimates, or optimization claims. | Paper attribution only. |
| [EvalAI](https://github.com/Cloud-CV/EvalAI), [paper](https://arxiv.org/abs/1902.03570) | Use explicit challenge phases, role separation, submission records, and public/private leaderboard views. | `app/api/app.py`, `app/execution_store.py`, `app/execution_arena.py` | Arbitrary code execution, remote workers, account platform, or EvalAI dependency. | Project/paper citation only; no code copied, images reused, or service integrated. |
| [Codabench](https://github.com/codalab/codabench), [paper](https://arxiv.org/abs/2110.05802) | Version challenge, policy, engine, scoring, worlds, and seeds as one portable reproducibility contract. | `app/execution_arena.py`, `app/execution_store.py`, `docs/EXECUTION_CHALLENGE.md` | Codabench bundles, workers, Docker submissions, or platform dependency. | Project/paper citation only; no code or bundle format copied. |
| [Reliable hidden leaderboards](https://arxiv.org/abs/1502.04585) | Public practice and protected final evaluation must be separate to reduce adaptation to hidden tests. | `app/api/app.py`, `tests/test_execution_arena.py` | Statistical guarantees beyond this prototype's explicit limits. | Paper attribution only. |
| [Metamorphic testing](https://arxiv.org/abs/2211.12003) | For simulations without one known optimum, paired relations and conservation invariants are stronger than a lone golden output. | `tests/test_simulation.py`, `tests/test_execution_arena.py`, `tests/test_execution_persistence.py` | A claim of exhaustive verification or proof over all worlds. | Paper attribution only; no testing code copied. |
| [OpenAI Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs) | Parse model responses through strict schemas, distinguish refusal/incomplete states, validate designer allow-lists, and validate every feedback evidence binding locally. | `app/execution_challenge_designer.py`, `app/execution_feedback.py`, `app/api/app.py`, `tests/test_execution_challenge_designer.py`, `tests/test_execution_feedback.py` | Unvalidated prose parsing, model-authored numeric worlds, hidden data before release, or an LLM-owned score/rank. | Official documentation citation; API use remains subject to OpenAI terms. |
| [OpenAI Evals](https://developers.openai.com/api/docs/guides/evals) | Freeze adversarial and expected cases so grounding is a repeatable regression gate, not one successful prompt. | `tests/fixtures/execution_feedback_eval_cases.json`, `tests/test_execution_feedback.py`, `tests/test_execution_challenge_designer.py` | A hosted eval dependency or live API requirement in CI. | Official documentation citation; local fixtures are project-authored. |
| [GPT-5.6](https://openai.com/index/gpt-5-6/) | Use the model only for bounded lesson intent and evidence interpretation; deterministic code stays authoritative for world construction, exchange, and scoring. | `app/execution_challenge_designer.py`, `app/execution_feedback.py`, `app/api/app.py`, `app/static/arena.js` | Model-generated prices, fills, metrics, ranks, numeric world parameters, or release state. | Product-page attribution; optional API use is subject to OpenAI terms. |
